# AGENT CORE PRINCIPLES

> Fundamental architecture and software engineering principles.
> Stack-agnostic. Applicable to any project, language, or platform.

---

## 1. LAYERED ARCHITECTURE

Code is always organized in layers with isolated responsibilities and unidirectional dependencies:

```
INTERFACES        → delivery (HTTP, WebSocket, CLI, mobile, UI)
APPLICATION       → orchestration (use cases, commands, queries, handlers)
DOMAIN            → pure business rules (entities, aggregates, events)
INFRASTRUCTURE    → external implementations (database, cache, APIs, queues)
```

**Absolute rule:** inner layers never depend on outer layers.
Domain does not know the database. Application does not know HTTP. Interface does not contain business logic.

---

## 2. DOMAIN LAYER IS THE CORE

- All business rules live in the Domain Layer.
- Domain imports nothing from frameworks, ORMs, HTTP libraries, or any external dependency.
- Entities, Aggregates, and Value Objects are rich objects with behavior — not anemic data structures.
- Each Aggregate has a root (Aggregate Root) that controls the lifecycle of the whole.
- Value Objects are immutable and self-validate on creation. Invalid values fail at construction.

---

## 3. EXPLICIT, NAMED BUSINESS RULES

Each business rule is:
- **Identified** — has a code or name (e.g. BR-001, `UniqueChipRule`)
- **Documented** — written in business language before any code
- **Tested** — covered by unit tests before implementation
- **Centralized** — never duplicated across layers

Standard format:

```
GIVEN [context/current state]
WHEN [action or event]
THEN [expected outcome]
AND [side effects or events raised]
```

---

## 4. EXPLICIT STATE MACHINES

Any entity that changes state over time has a documented state machine:

- Possible states are enumerated and finite.
- Valid transitions are declared explicitly (from → to, trigger, allowed role).
- Invalid transitions throw domain exceptions — never silent.
- Terminal states allow no further transitions.

State machines live in the Domain Layer. No outer layer decides the next state directly.

---

## 5. DOMAIN EVENTS

Business facts are communicated via immutable events:

- An event represents something that **already happened** — name is always past tense (`AnimalRegistered`, `OwnershipTransferred`).
- Events are raised by the entity or aggregate, never directly by the application layer.
- Events are the communication mechanism between contexts — never direct calls between domains.
- Every event carries: unique identifier, timestamp, origin (context/tenant), and relevant payload.
- Events are immutable and traceable. Never edited or deleted.

---

## 6. HISTORY IS IMMUTABLE

Any history record (audit, health, location, transactions) is never edited or removed.

- Corrections are new records referencing the corrected record.
- Audit logs cover every write: who, when, what, before and after.
- Audit data has a defined and enforced retention policy.

---

## 7. IDENTITY AND VALUE OBJECTS

- Entity identifiers are generated by the domain (not the database).
- Value Objects encapsulate validation and formatting — an invalid CPF never exists as an object.
- Business-readable identifiers (e.g. `SSO-2024-00001`) coexist with internal UUIDs.
- Sensitive data (CPF, email, phone) is masked in logs and API responses.

---

## 8. ACCESS CONTROL (RBAC + ABAC)

- Roles are defined in the domain with business semantics (Inspector, Owner, Admin) — not generic strings.
- Permission matrix is documented explicitly per action × role.
- Beyond role (RBAC), access may depend on contextual attributes (ABAC): e.g. clinic only sees animals currently in care.
- Authorization checks happen in the Application layer, before any domain operation.
- In multi-tenant systems, tenant isolation is verified on **every** request — no exceptions.

---

## 9. USE CASES DOCUMENTED BEFORE CODE

Each use case has:
- **Actor** — who executes
- **Preconditions** — what must be true beforehand
- **Main flow** — happy path step by step
- **Alternate flows** — variations and exceptions
- **Rules applied** — which BRs are verified
- **Events raised** — what happens after execution

An undocumented use case must not be implemented.

---

## 10. EXPLICIT API CONTRACTS

- Every API has explicit versioning from day one (`/v1/`, `/v2/`).
- Authentication and tenant identification are mandatory on every request.
- Error responses follow a single structured pattern (never random strings).
- Pagination, sorting, and filters follow a uniform convention.
- Endpoints reflect the domain — not the database. Resources are named by business concept.

---

## 11. SECURITY AS REQUIREMENT, NOT FEATURE

- Strong authentication (MFA) for elevated-privilege roles.
- Session TTL defined per role — admin sessions expire faster.
- Rate limiting per user and per IP on every public API.
- Sensitive data encrypted at rest and in transit.
- All external communication uses TLS. No exceptions.
- Data legislation compliance (LGPD, GDPR, etc.) is a functional requirement from day one: consent, portability, erasure, anonymization.

---

## 12. TEST-DRIVEN DEVELOPMENT (TDD)

Test pyramid:

```
E2E (5%)          → full end-to-end flows
Integration (20%) → use cases + infrastructure
Unit (75%)        → domain layer (entities, rules, state machines)
```

- Domain tests are written **before** implementation.
- Standard test format: GIVEN / WHEN / THEN.
- Minimum Domain Layer coverage: 90%+.
- A test must fail for exactly one reason.
- Tests do not depend on real database, network, or filesystem.

---

## 13. AI / AGENT INTEGRATION

When the system integrates LLMs or AI agents:

- The AI agent operates **inside** the Application Layer — never accesses Domain or Infrastructure directly.
- AI reads only. Never writes directly to the database.
- Every AI-generated suggestion requires human confirmation before persistence.
- Agent access is restricted to the minimum required for the task (least privilege).
- Every LLM call is logged: input, output, model, timestamp, triggering user.
- AI responses are always marked as suggestions — never as definitive truth.
- Personal and sensitive data is never sent to the LLM. Only aggregated or anonymized data.

### OWASP Agentic Applications:2026 (ASI01–ASI10)

For autonomous or tool-using agents, also enforce `rules/03-security/OWASP-AGENTIC-2026.md`:

- ASI01 Agent Goal Hijack — untrusted content cannot override goals
- ASI02 Tool Misuse — allow-listed tools, scoped permissions
- ASI03 Identity and Privilege Abuse — scoped agent credentials
- ASI04 Agentic Supply Chain — verify MCP servers, skills, rule packs
- ASI05 Unexpected Code Execution — no auto-run of agent-generated code on prod
- ASI06 Memory and Context Poisoning — versioned rules; no agent self-write to memory
- ASI07 Insecure Inter-Agent Communication — authenticate agent messages
- ASI08 Cascading Failures — circuit breakers, step limits
- ASI09 Human-Agent Trust Exploitation — human confirm for destructive actions
- ASI10 Rogue Agents — monitoring, kill switch, audit trail

Web/API security: **OWASP Top 10:2025-aligned** rules in `rules/03-security/` (current through 2026).

## 14. MINIMUM PROJECT GLOSSARY

Every project maintains a glossary of domain terms. This ensures ubiquitous language across engineering, business, and AI agents.

The glossary defines:
- Business terms and precise definitions
- Enums and their possible values
- Acronyms and abbreviations used in code

Code using a domain term uses **exactly** the same name as the glossary — no translations, abbreviations, or creative synonyms.

## CHECKLIST FOR ANY NEW PROJECT

Before writing the first line of code:

- [ ] Are layers defined and responsibilities documented?
- [ ] Are main entities and aggregates identified?
- [ ] Are Value Objects listed with their invariants?
- [ ] Are business rules written in GIVEN/WHEN/THEN language?
- [ ] Are state machines documented with valid transitions?
- [ ] Are access roles and permission matrix defined?
- [ ] Are domain events catalogued?
- [ ] Are use cases documented with actor, flow, and preconditions?
- [ ] Is the API contract sketched?
- [ ] Does the domain glossary exist?

If any item is blank, the agent **must ask before implementing** — never assume.
